## Summary

HTTP tools combine **saved tool definitions** (your organization’s database) with **live call context** at runtime. Optional **integration-backed response caching** is a **roadmap** item; it is **not** enabled in the product today and there is **no** cache-related environment toggle.

Read the full authoring story in the [HTTP API tool guide](/voice-agent/tools/http-api) — especially **Storage model** and **Template resolution order**.

## Today (operators)

| Surface | Where it lives |
|---------|----------------|
| Tool URL, headers, parameters, body template, response mapping, credentials | **Org database** (saved with the tool) |
| `{{…}}` values during a live call | **Runtime** — merged arguments + call / conversation context from the session |
| **Test API Call** sample JSON and call-context editor | **Browser only** — not written to the org DB |

## Runtime extension point (no env flags yet)

There is **no** environment variable or toggle that turns on HTTP response **execution** caching. A small **`http_tool_cache_policy.py`** module centralizes policy constants and parsing for **WE-01-DATASTORE-INTEG** (do not add silent runtime caches).

**Policy API (runtime off, draft on):** `GET` / `PUT` `/api/v1/organizations/http-integration-cache-policy` (authenticated, org-scoped) return effective `cache_enabled: false`, deferral, `implementation_status`, `policy_schema_version`, merged **`stored_preferences`** (`cache_enabled_when_shipped`, optional `ttl_seconds`), and **`policy_audit`** (rolling tail when drafts change — actor id + timestamp). **PUT** persists draft fields and appends an audit row only when values change; HTTP tool calls remain live-only until runtime cache ships. Successful **PUT** also emits PostHog `http_integration_cache_policy_updated` when telemetry is configured. See [org datastore design](/integrations/http-tool-org-datastore-design).

### Governance and audit (draft policy era)

Use this section for security reviews until **per-integration** execution caches ship.

- **Data minimization:** The policy API stores only org intent (`stored_preferences`), a versioned JSON envelope, and **`policy_audit`** rows (timestamp, actor identifier, and the draft fields saved). It does **not** store HTTP response bodies, upstream URLs with secrets, or voice payloads.
- **Immutability:** Audit entries are append-only in configuration JSON (trimmed to a fixed maximum length server-side). Editors surface recent saves under **Platform settings**.
- **Access:** Mutations require an authenticated session with your normal org authorization path (same as other org configuration APIs). Audit **`actor_provider_id`** identifies the saver for accountability—not an end-customer phone number or email.
- **Telemetry:** When PostHog is configured, **`http_integration_cache_policy_updated`** records organization id and coarse flags (for example whether TTL was set); it must not include credentials or integration secrets.
- **Separation of concerns:** Call-analytics **PII redaction** for QM exports and call detail (MK-01) is unrelated to integration-cache drafts—continue to treat analytics governance independently.

## Roadmap (WE-01-DATASTORE-INTEG)

When implemented, expect:

- **Org-admin** toggles per integration: cache on/off, TTL, PII class, audit of policy changes (today you have **org-wide draft** prefs + **`policy_audit`** only).
- **Authoring** UI: explicit “local only” vs “also persist/cache via integration X” — no silent dual-writes.
- **Runtime**: read-through or short-lived cache only when policy allows; fallback to live API on failure.
