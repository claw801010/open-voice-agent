HTTP API tools allow you to attach external REST API calls directly to workflow nodes, enabling your voice agents to call any internal or external system during live conversations based on LLM judgment and your prompts. This works similar to function calling on any agentic platform and is 100% open source and fully customizable.

## What is an HTTP API Tool?

An HTTP API Tool is a REST API definition that the LLM can invoke at runtime.

**Typical use cases:**
- Call your own backend endpoints
- Trigger n8n automations
- Sync data with a CRM
- Fetch data from external APIs (weather, pricing, availability, etc.)
- Write/update/read data using Rest API

**The LLM decides:**
- which tool to call
- when to call it
- what parameters to send

**All based on:**
- Your prompts: node-level instructions in simple english (or any language)
- Tool name
- Tool description
- Parameter definitions

## Defining an HTTP API Tool

### 1. Tool Name

- Must be clear and action-oriented.

- **Examples:** `capture_lead_interest`, `fetch_weather`, 'create_crm_contact', etc

### 2. Tool Description


- Extremely important
- This is how the LLM decides **when** to use the tool. 
- Write it in plain, explicit English.

**Bad:**
"API to capture data"

**Good:**
"This tool is to capture interest. Use this tool when the user clearly expresses interest in the product or wants to be contacted"

![Tool Description Example](/images/tool%20description.png)

### 3. Endpoint Configuration

- Full URL (**must include `http://` or `https://`**)
- Supports **REST** methods
- **Templated URLs:** you may include `{{path.to.value}}` in the path or query, for example `https://api.example.com/orders/{{order_id}}`. The editor accepts these while you are authoring; the URL is resolved at runtime. See [Template resolution order](#template-resolution-order) below. In the dashboard, the **Endpoint URL** field includes the same **grouped variable picker** as parameters and headers so you can insert system, conversation, custom, and tool keys at the caret without retyping. Each picker opens a searchable list (filter by substring) so long variable sets stay easy to navigate.
- **Picker insert mode:** **Replace selection / insert at caret** replaces a highlighted range, or inserts the chosen `{{…}}` at your text cursor when nothing is selected (it does not wipe the whole URL or JSON). **Append at cursor** inserts after the end of the selection (or after the caret when collapsed). **Custom flow variables** are stored **per HTTP tool** in the browser (localStorage), with a one-time copy from the legacy global key when you first open a tool after upgrading.
- **Picker hints:** built-in **System defaults** and **Conversation state** rows show a short description under each `{{…}}` token; the filter box matches both the token and that hint (for example search `E.164` to find `{{caller_number}}`). **Test payload** JSON for **Test API Call** is also stored **per tool** in the browser (`tool-http-test-payload:{toolUuid}`) and is not part of the saved tool definition on the server.
- **Raw Code** tab uses the same grouped variable insert control under the editor (place the caret, then pick a token). **Test call context** sample JSON and the call-context **Form / JSON** tab choice are also remembered **per tool** in the browser (`tool-http-call-context-json:{toolUuid}` and `tool-http-call-context-editor-tab:{toolUuid}`), migrating once from the legacy global `tool-http-call-context-json` key when present.

<Note>Common mistake: forgetting https:// in the URL. Templated URLs must still start with `http://` or `https://`.</Note>

### 4. Authentication & Headers

- Add custom authentication
- Add custom headers (values support the same `{{…}}` paths as the URL and body; see [Template resolution order](#template-resolution-order))
- Works with internal services and third-party APIs

### 5. Parameters

Each parameter must have:
- Name
- Type
- Description
- Required/Optional flag

**Parameter descriptions matter more than types.**

Guidelines:
- Start with string parameters when possible
- Be explicit in what the value represents
- Mark only truly mandatory fields as required

Example:
- interest (string):
  "Set to true if the user clearly shows intent to buy or wants follow-up. Otherwise false."

## Attaching Tools to Workflow Nodes

- You can attach **multiple tools to a single node**
- All the tools that you have created will be available for selection in the node
- Tools are only callable when attached to that node
- The LLM will choose which one to call

Inside the node, guide the LLM using **simple English instructions**.

Example:

"If the user shows interest in speaking to sales or wants a callback, immediately call the capture_lead_interest tool and set interest to true."
This instruction is often the **deciding factor** for correct tool usage.

## Tool Invocation Logic (How the LLM Thinks)

The LLM considers:
- User's spoken intent
- Node prompt instructions
- Tool name and description
- Parameter descriptions

If these align clearly, the tool is called automatically.

Poor naming or vague descriptions lead to:
- Missed tool calls
- Wrong parameters
- Hallucinated values

## Template resolution order

`{{…}}` placeholders in the **body template** (default JSON), **LLM tool parameters**, **endpoint URL**, and **header values** (including structured header rows) are resolved in a consistent way:

1. The tool starts from **LLM arguments** and fills in **parameter `value_template` defaults** where the model did not supply a value.
2. The **body template** is merged in as defaults and combined with those arguments to form the final JSON payload (for methods that send a body or query string).
3. The **URL** and all **header** values are then resolved using that **merged** data **plus** the current **call and conversation context** (system fields, `conversation.*`, your workflow template variables, and similar paths described in [Context and variables](/core-concepts/context-and-variables) and [Template variables](/voice-agent/template-variables)).

So a key that exists only in the **body template** can still be used in a **header** (for example a trace or customer id) as long as the path matches.

In the dashboard, use **Test API Call** with a sample **test payload** and optional **call / conversation context** JSON. In the **Form** view, **Preset path** includes system and conversation paths plus a merged group (for example **“From your flow (custom & tool keys)”**) built from this tool’s custom template keys, parameter names, and response-mapping key names, so you can build sample context without retyping every dot path; the preset and template pickers are **searchable** (type to filter). After a successful test, the UI can highlight any `{{…}}` paths that were not found in those samples, including paths used in the **URL** or **headers**.

**Response mapping** (Simple and Advanced) uses **plain dot paths** into the **HTTP response body** (for example `data.id` or `data.items.0.id`). This is different from the `{{…}}` system used for the URL, parameters, and call context. The insert control lists common response shapes; **Auto-map from latest response** (after a successful test) is still the fastest way to build mappings.

In **Test API Call** → **Call / conversation context (test only)**, use **Add missing sample values** to merge the standard app default shape (system, `conversation`, `initial_context`) into your sample without overwriting fields you already set. **Reset sample context** replaces the full JSON with the default. Browser only (not stored on the tool).

In the **Form** tab of that editor, **Add missing preset rows** adds one row per path shown in **Preset path** (including merged flow paths) that is not already in the form, and fills values from the same default sample when available—useful when you want to edit many paths as rows instead of JSON only. Choosing a **Preset path** on an existing row while the **Value** cell is empty also applies that default sample value when the path exists in the standard shape (hover group headers in the picker for what each group contains).

## Storage model

This section clarifies **what is stored where today** versus a **future** optional integration cache (**WE-01-DATASTORE-INTEG** in the execution roadmap). It helps operators and compliance reviewers reason about HTTP tools without confusing the dashboard sample editor with production persistence.

### Org-local and runtime (today)

- **Tool definition** (URL, method, headers, parameters, body template, response mapping, credentials reference, and so on) is saved with your **organization** through the product APIs and backing database—this is the **source of truth** for how the agent should call the HTTP endpoint.
- **Live `{{…}}` resolution** uses the **merged tool arguments + body defaults** and the **current call and conversation context** from the active voice/session pipeline (system fields, `conversation.*`, `initial_context.*`, and workflow template variables as documented in [Context and variables](/core-concepts/context-and-variables)). That context is **runtime data**, not something you edit as a separate “datastore” row in this HTTP tool screen.
- **Test API Call** → **Call / conversation context** sample JSON and related **browser-only** conveniences (including **Add missing sample values** / **Reset sample context**, the **Form** tab presets, and **Add missing sample keys (JSON)** on the **JSON** tab) are stored **in this browser** so you can iterate on tests without changing the saved tool. They are **not** a substitute for production call context and are **not** written to the org database as call logs.

### Integration-backed cache (roadmap, not shipped)

A future package may add an **explicit**, **org-admin–controlled** option to **read through** or **short-lived cache** normalized slices of responses from **integration-backed** connectors (TTL, PII class, audit). Until that ships:

- There is **no** automatic replacement of org-local records with third-party caches.
- HTTP tools continue to behave as **live REST calls** (plus your saved definition and runtime context), unless you implement your own persistence downstream of the API you call.

When **WE-01-DATASTORE-INTEG** is implemented, product docs and the HTTP tool UI will describe the **authoring contract** (“local only” vs “also cache via integration X”) and **admin policy** in one place so authors never get ambiguous dual-writes.

Operator-focused policy summary (integrations hub): [HTTP tools — org data and cache policy](/integrations/http-tool-data-policy). Org-scoped **read stub** for cache posture: `GET /api/v1/organizations/http-integration-cache-policy` (see [design doc](/integrations/http-tool-org-datastore-design)).

## Key Best Practices
- Name tools clearly
- Write detailed, action-based descriptions
- Keep parameters simple at first
- Always include http/https in URLs; templated paths still need a valid `https://` prefix in the editor
- Use plain English in node instructions
- Attach only relevant tools to each node

**Well-defined tools + clear prompts = reliable, production-grade voice agents.**
